- [ ] **§10 非目标**至少列了 3 个本期不做的事。
- [ ] **§13 关联**至少包含 1 个 Kanban 卡 ID（或显式豁免理由）。

## 内容门

### §1 产品背景

- [ ] 引用 Positioning 的 WHY（link，非重写）
- [ ] 没有"行业趋势"——只写用户/产品上下文
- [ ] 至少识别 1 个具体的触发问题（commit / spec / 反馈）

### §2 目标用户

- [ ] 区分现有用户和未来用户
- [ ] 引用 Positioning 的 WHO（link，非重写）
- [ ] 没有"给所有用户"——至少 1 个具体角色

### §3 用户故事

- [ ] 每个 US 都有"作为 / 想要 / 以便"结构
- [ ] 每个 US 有 checkbox 验收标准
- [ ] 至少 1 个 US 端到端可测

### §4 功能需求

- [ ] 每个 FR 有实现级细节（不写代码，但指定行为）
- [ ] FR 与 US 一一对应（或有显式豁免）

### §5 非功能需求

- [ ] 6 个维度都已处理（性能 / 安全 / 隐私 / 可扩展 / 可观测 / 可回滚）——允许"无"+ 理由
- [ ] 引用 Positioning 的 WHY NOW（link，非重写）

### §6 数据迁移

- [ ] 改 schema 的话强制——备份 / 转换 / 干跑 / 验证都在
- [ ] 不适用 → "无"+ 理由

### §7 数据可观测性

- [ ] 项目产生可查询数据的话强制——数据流 + 查询已列
- [ ] 不适用 → "无"+ 理由

### §8 前端改动

- [ ] 改 API 或加 UI 的话强制——组件 / UX 文案 / TZ 处理
- [ ] 不适用 → "无"+ 理由

### §9 风险

- [ ] 至少 1 个风险 + 缓解
- [ ] 每个风险都有等级

### §10 非目标

- [ ] 至少 3 个显式非目标
- [ ] 镜像 Positioning 的 ANTI-POSITIONING（link，非重写）

### §11 验收标准

- [ ] 全程 checkbox 格式
- [ ] 覆盖功能 / 性能 / 测试 / 迁移 / 回滚

### §12 可观测性需求

- [ ] §12.1 列了新增事件（触发 / metadata / 用途 / 优先级）
- [ ] §12.2 列了复用事件（metadata 加字段或不变）
- [ ] §12.3 schema 改动显式（或声明"无 schema 改动"）
- [ ] §12.4 验收标准有 100% 覆盖率目标
- [ ] §12.5 隐私考量（如涉及敏感字段）

### §13 关联

- [ ] Kanban 卡 ID 在
- [ ] 前序 PRD / spec 链接了（如存在）

---

## 质量门

强烈推荐（不强制，但跳过通常意味着 PRD 还没准备好）。

- [ ] **PRD 总长度合理**——多数项目 < 2000 行。更长则拆成多个 PRD。
- [ ] **§3 / §4 / §12 交叉引用一致**——每个 US 都能追溯到 FR 追溯到可观测事件。
- [ ] **没有编造的指标**——§5 / §11 每个数字都有依据（Positioning、前序实验、行业标准）。
- [ ] **强制章节没有 "TBD"**——§6 / §7 / §12 的 TBD 意味着你没决定，会卡下游。

---

## 自检问题

1. **从没看过这个 codebase 的聪明工程师能实现这个 PRD 吗？** 如果不能，缺什么上下文？
2. **§10 非目标列出的是团队想要做的事吗？** 如果不是，scope creep 即将发生。
3. **§12 可观测性够完整吗，admin dashboard 上线第一天就有数据？** 如果你盲发，你忘了埋事件。
